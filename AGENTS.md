# Инструкции для AI-агентов (корневой `AGENTS.md`)

**Прочитайте этот файл в начале каждой задачи.** В нём — обязательный порядок работы и границы, без привязки к стеку. Смысл продукта, структура каталогов и соглашения репозитория — **в `README.md`** и в остальной документации репо, **если она есть** — **её тоже нужно прочитать**, когда она относится к задаче.

---

## 1. Сначала прочитать

1. **`README.md`** — обязательно: назначение проекта, сборка, запуск, тесты, дальнейшие ссылки.
2. **Все файлы, которые относятся к текущей задаче** (исходники, конфиги, тесты, CI) — **нужно** открыть и учесть; пути **нельзя** угадывать — их **нужно** взять из `README` и фактической структуры репозитория.
3. **Если в репо есть** каталоги или файлы **`docs/`**, **`CONTRIBUTING`**, **`.cursor/rules`**, вложенные **`AGENTS.md`** — **прочитайте** релевантные разделы **до** правок, когда задача их затрагивает.

Если в репозитории задан **порядок версионирования и коммитов** (в `docs/`, `CONTRIBUTING`, правилах) — **соблюдайте его**; он **вышестоящий** относительно абзацев ниже.

---

## 2. Что отслеживается в git

- **`[.gitignore](.gitignore)`** (и аналоги) — **нельзя** коммитить перечисленные там пути **без явного прямого запроса пользователя**.
- **Сверяйтесь** с `README` и с деревом файлов **после** `git clone` — **не** предполагайте наличие каталогов, которых **нет** в remote.

*Журнал сессий / история* (`docs/HISTORY.md` и т.п.): **если** **в документации репозитория или в правилах** прямо требуется вести такой журнал — **нужно** дописывать запись **по завершении задачи**; **если** в `.gitignore` этот файл исключён из git — **всё равно** ведите его **локально**, пока требование действует.

---

## 3. Обязательно делать

- **Соблюдать** стиль кода репозитория: форматирование, именование, модули, существующие абстракции.
- **Делать минимальный дифф:** закрывать **только** сформулированный запрос; **нельзя** рефакторить и менять соседний код **без** необходимости для этой задачи.
- **Соблюдать согласованность:** новые публичные API, переменные окружения, схемы, флаги CLI — **вводить по тем же правилам**, что и остальной проект, и **документировать** рядом, **если** так принято в репо.
- **Версия и публикация:** **если** в репозитории есть соглашение о номере релиза и коммитах (файл `VERSION`, `package.json`, `pyproject.toml`, теги, раздел в `docs/`) — **нужно**:
  - определить единый источник правды о версии **по коду и доке репо**;
  - **поднять** версию **по правилам** этого репозитория;
  - **оформить** сообщение коммита **в требуемом формате**; **если** формата **нет** — первая строка **обязана** чётко говорить, **что** сделано и **зачем**;
  - выполнить **`git add` / `commit` / `push`**, **когда** есть доступ к сети и к remote — **нельзя** односторонне поручать пользователю коммит, **если** вы можете выполнить его сами.
- **Документация:** **нужно** **обновить** `README` и/или `docs/`, **если** изменилось **пользовательское** поведение, CLI, конфиг или контракт API.

---

## 4. Запрещено

- **Запрещено** поднимать приложение, долгосрочные сервисы или **полный** интеграционный стенд **в среде агента** «для проверки», **если** пользователь **не** попросил это явно **или** **не** сказано иное **в** правилах репо. **Разрешены** линтеры, статические проверки и **любые** команды, **которые пользователь** указал **прямо**.
- **Запрещено** коммитить секреты, ключи, токены, прод-данные; **разрешены** только шаблоны (например `.env.example`), **как** в репо.
- **Запрещено** форсить в git пути из `.gitignore` **без** явного запроса пользователя.

---

## 5. Ориентиры в незнакомом репо

| Вопрос | Где **нужно** искать |
|--------|----------------------|
| Вход в приложение | `README`, `package.json` / `main` / `cmd/` / `app/` — **по** соглашениям стека |
| Конфигурация | env-шаблоны, `config/`, **по** доке фреймворка |
| Тесты и CI | `.github/`, `.gitlab-ci.yml`, `Makefile`, `scripts/` — **сначала** `README` |
| Контракт с пользователем | `README`, OpenAPI, `docs/`, changelogs |

---

## 6. Резюме

**Сначала** — `README` и **все** применимые правила репозитория, **затем** — правки **строго** в существующем стиле. **Версия и коммит** — **по требованиям** репо, **с push при возможности**. **Нельзя** раздувать scope **и** **нельзя** запускать у себя целую инфраструктуру без прямой команды.
